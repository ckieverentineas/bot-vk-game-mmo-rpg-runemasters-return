# Plan — масштабирование Runemasters Return

## Цель

Подготовить проект к двум параллельным скоростям роста:

- быстро добавлять новый контент мира, рун и боевых сценариев;
- быстро и безопасно выкатывать обновления без ручной охоты за разрозненными точками правды.

## Что уже закрыто

- [x] модульная архитектура с разделением `domain` / `application` / `infrastructure` / `transport`;
- [x] единый каталог команд и keyboard-first транспорт;
- [x] адаптивная сложность без ручного выбора уровня угрозы;
- [x] единая загрузка игрока для use-case слоя;
- [x] единая recovery-логика зависших боёв;
- [x] commit-based versioning и релизный preflight.

## Ближайший план

### Этап 1. Стабильное основание

- [x] убрать мёртвые transport-команды и связанный разветвлённый код;
- [x] вынести повторяющиеся проверки игрока в общий helper;
- [x] вынести повторяющуюся recovery-логику боя в отдельный application helper;
- [x] защитить боевые награды, расход осколков и активные бои от повторных команд и underflow-сценариев;
- [x] добавить release preflight для корневой документации.

Критерий успеха:

- новый use-case подключается без копирования ошибки `player_not_found`;
- любой recovery активного боя меняется в одном месте;
- повторный входящий `атака`, `создать` или `сломать` не должен дублировать награды и не должен уводить ресурсы в минус;
- перед выкладкой достаточно прогнать `npm run check` и `npm run release:preflight`.

### Этап 2. Ускорение поставки контента

- [x] ввести явные контентные контракты для новых биомов, мобов, рун и событий;
- [x] покрыть чистые доменные функции тестами для контентных правил и боевых формул;
- [x] добавить автоматическую валидацию контентных сидов в `check`, `seed` и `release:preflight`;
- [ ] отделить контентные таблицы от transport-описаний, чтобы дизайнерские правки не трогали VK-слой.

Критерий успеха:

- новый биом или набор мобов добавляется без изменений в `vk/handlers`;
- новая руна или эффект не требует правок в Prisma-репозитории вне нужного маппинга;
- формулы и контентные ограничения проверяются автотестами и контентной валидацией.

### Этап 3. Быстрая доставка обновлений

- [x] оформить единый релизный чек-лист для локальной поставки и CI;
- [x] автоматизировать генерацию changelog/release summary из истории изменений;
- [x] добавить smoke-проверки для ключевых пользовательских сценариев: старт, бой, руны, обучение.
- [x] усилить player-facing copy в онбординге, обучении и бою, чтобы у каждого ключевого экрана был явный следующий шаг.

Критерий успеха:

- релиз можно проверить и описать без ручного обхода всего проекта;
- пользовательские документы не забываются при функциональных изменениях;
- базовые игровые сценарии валидируются до публикации обновления.

### Этап 4. Масштабирование систем игры

- [x] собрать единый paged rune hub вместо неудобного one-by-one browsing;
- [x] перестроить ранний battle UX в event-first формате с явным CTA после результата;
- [x] сгладить early game через усиление стартового героя, retune первых мобов и мягкое правило инициативы;
- [x] ограничить случайные high-rarity rune drops на низких уровнях;
- [ ] активные и пассивные навыки рун как реальные боевые действия, а не только контентные описания;
- [ ] дополнительные боевые действия и статусы;
- [ ] предметы, крафт, события, квесты и сезонные режимы;
- [ ] DTO-слой application-контрактов при росте transport-каналов.

Критерий успеха:

- новые системы добавляются через отдельные модули и маленькие use-case'ы;
- transport остаётся тонким и не хранит игровые правила;
- игрок может быстро найти нужную руну в коллекции без утомительного листания по одной записи;
- ранний бой остаётся понятным, читаемым и не ломает мотивацию в первые сессии;
- релиз новых систем не ломает текущую базовую петлю игры.

## Метрики готовности

- `npm run check` проходит стабильно;
- `npm run content:validate` отлавливает битые сиды и баланс до релиза;
- `npm run release:preflight` подтверждает готовность корневой документации;
- `CHANGELOG.md`, `README.md`, `PLAN.md` обновляются синхронно при пользовательских изменениях;
- доменная логика растёт за счёт новых модулей и helper'ов, а не за счёт одного большого handler'а.

## Правило обновления плана

При каждом крупном изменении архитектуры, релизного процесса или пользовательской игровой петли этот файл должен обновляться одновременно с `README.md` и `CHANGELOG.md`.
